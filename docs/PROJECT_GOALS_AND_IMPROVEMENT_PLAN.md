## 1. ビジョン（最終ゴール）

**「医療・介護・福祉の対面現場で、言語の壁を"存在しないもの"にする道具」**

診察室・訓練室・面談室でポケットから取り出し、10秒以内に会話を開始できる。専門用語・施設名・人名を誤らず、話す→見せる→聞かせるの各動作が1タップで完結する。マニュアルを読まなくても、初見の相手（患者・利用者・家族）に画面を見せるだけで意思疎通が成立する——これを **v3.0 の達成状態** と定義する。

BridgeTTSは汎用翻訳アプリ（Google翻訳等）と競争しない。**「専門領域の対面会話」に特化**することが存在理由である:

| 観点 | 汎用翻訳アプリ | BridgeTTS |
|---|---|---|
| 専門用語 | 文脈不明で誤訳しがち | 領域プロンプト＋ユーザー辞書で固定 |
| 施設名・人名 | 誤変換・直訳 | ユーザー辞書で常に正しい表記・英語 |
| 対面UI | 汎用（小さい表示） | ターン制・大型表示・タップ読み上げ |
| データ | クラウド送信・履歴収集 | 端末内完結（送信はOpenAI翻訳APIのみ） |
| 配布・改善 | ベンダー任せ | 自前配布（AltStore）＋アプリ内エラー報告で即改修 |

## 2. 対象ユーザーと利用場面

- **一次**: 医療・介護・福祉職（リハ職・心理職・看護・介護・相談員）。外国人患者・利用者・家族との問診、訓練指導、生活相談、サービス説明。
- **二次**: 同僚・施設内の他職種（ソースURL共有で配布）。
- **三次**: 接客・日常会話（翻訳モード「日常会話」でカバー。ただし設計判断は常に一次ユーザー優先）。

## 3. 成功指標（KPI — v3.0判定に使う）

1. **速さ**（リアルタイム性。計測機能をv2.7で内蔵）:
   - **訳出開始**（発話の区切り検出→最初の訳語が画面に出るまで）: 中央値 **700ms以内**
     （内訳目標: 区切り検出=デバウンス154〜346ms ＋ API初トークン350ms以内）
   - **逐次表示**: 発話中もストリーミングで訳が更新され続け、発話終了時点で訳の大半が表示済みであること
   - **訳の確定**（発話終了→全文が安定表示）: 中央値 **1.2秒以内**・95パーセンタイル **2.0秒以内**
   - 参考水準: 人間の同時通訳の遅延3〜5秒／研究評価(IWSLT)の遅延制約 約2秒／市販翻訳機の公称値は応答0.2秒・翻訳完了3秒未満（Timekettle）。本目標の「訳出開始700ms・確定1.2秒」はこれらの実用水準より厳しい設定であり、体感は「話し終わる前に訳が出ている」状態を狙う
2. **操作**: 起動→会話開始 **2タップ以内**、会話中の全操作（開始/停止/再生/見せる）**1タップ**
3. **品質**: 医療評価ケース50件で **致命的誤訳0件**・要修正10%以下（評価ハーネスで判定）
4. **信頼性**: 30分連続使用でクラッシュ・無応答・音声途絶 **0件**、既知バグ（error-report Issue）常時0維持
5. **満足**: 同僚テスト（3名以上）で「また使いたい」全員肯定＋簡易SUS 80点相当

### 4.2 非機能要件

- **対応環境**: iOSネイティブ（iOS16+、AltStore配布）を主、iOS Safari PWA / Android Chrome PWAを従とする
- **プライバシー**: 会話・辞書・設定は端末内のみ。外部送信は翻訳テキスト（OpenAI API）とユーザーが明示送信するエラー報告のみ。この方針をUIとREADMEに明記し続ける
- **保守性**: 変更は追加中心・最小差分。コア（認識/翻訳/TTS）は検証なしに触れない。スモークテスト常時グリーン。app.jsの責務分割は「挙動を変えない範囲で」v2.6以降段階的に
- **運用**: `v*`タグ→自動リリース→AltStore配信の完全自動を維持。error-report Issueは次リリースまでに処置

## 5. スコープ外（やらないこと）

いたずらな機能追加を防ぐため、以下は**明示的に作らない**:

- 日英以外の言語対応（v3.0までは品質を日英に集中）
- アカウント・ログイン・クラウド同期（エクスポート/インポートで代替）
- 会話履歴の外部共有・SNS連携
- 音声通話・遠隔通訳（対面特化）
- App Store配布（AltStore運用で十分。TestFlightはv3.0達成後に再検討）

新機能の採否は「§3のKPIのどれかを改善するか？」で判定する。改善しないものは追加しない。

### v2.7 レイテンシ設計（確定済み・実装順）

現行アーキテクチャ（蓄積テキストの再翻訳＋ストリーミング表示）はGoogle翻訳本番と同方式（re-translation）であり変更しない。以下を実装順に投入し、各段でF12の実測値を確認する:

1. **F12 レイテンシ計測**: トリガー→送信→初チャンク→完了の4点を毎回記録し、中央値/95pを設定画面に表示（全施策の前提）
2. **プロンプトキャッシュ確実化**: システムプロンプトの静的部分（基本ルール＋領域指示）を先頭・1024トークン超に整え、OpenAIの自動キャッシュ（プリフィル遅延最大80%減・入力費90%減）を常時ヒットさせる
3. **接続ウォームアップ**: 録音開始時にapi.openai.comへ軽量リクエストを送りTLS/HTTP2接続を先行確立（初回翻訳の100〜300ms削減）
4. **増分翻訳**（品質影響ありうるためF11で50ケース比較後に投入）: 確定済みセグメントの訳を凍結し、能動文のみ翻訳対象にして確定時間を文長一定（約1秒）に保つ
5. **投機的先行翻訳**（4まででKPI未達の場合のみ）: デバウンス途中で先行リクエストを発射し、テキスト変化時はAbortControllerで破棄

遅延予算: 区切り検出154〜346ms＋接続0ms＋初トークン100〜300ms → 訳出開始300〜650ms（目標700ms内）

## 7. 開発フロー（一般的なアプリ開発サイクルとの対応）

一般的なフロー **企画→要件定義→設計→実装→検証→リリース→運用→振り返り** を、このプロジェクトでは次のように回す:

1. **企画/要件**: 本文書。機能の追加・変更はまず§4の表とKPIに照らす（合わないものは§5へ）
2. **設計**: 変更前にCLAUDE.md「触れてはいけない核」を確認し、追加中心・最小差分の設計をとる。高リスクはfresh-contextレビュー
3. **実装**: ブランチ→PR。関連する変更は1つのPRに束ねる
4. **検証**: node --check／Playwrightスモーク（ローカル+CI）／v2.6以降は評価ハーネス。実機確認が必要な変更はリリースノートに明記してユーザーに委ねる
5. **リリース**: 版数整合→CHANGELOG確定→`v*`タグ→自動配信（AltStoreに即時反映）
6. **運用**: アプリ内エラー報告→Issue→次リリースで処置。ダッシュボードで状態監視
7. **振り返り**: リリースごとに本文書§4.1の状態列とHANDOFF.mdを更新。教訓はAGENT_PLAYBOOK/HANDOFFへ

## 8. 技術方針（現行アーキテクチャの維持事項）

- WKWebView＋Webアプリ共通コア（PWA/ネイティブの二重保守を避ける）
- 翻訳は端末→OpenAI直（gpt-4.1-nano）。品質はプロンプト＋辞書で作り込む（モデル変更は評価ハーネスで比較してから）
- 保存は端末localStorage（可搬性はF10で担保）
- 配信はGitHub Actions完全自動（人手の配布作業ゼロ）
