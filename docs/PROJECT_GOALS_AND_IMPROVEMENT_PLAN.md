# Bridge-TTS Codex: 現状の問題点、改善策、ゴール

作成日: 2026-05-18

## プログラムのゴール

Bridge-TTS Codex は、日本語と英語の会話をスマートフォンで素早く橋渡しする PWA を目指す。最終的には、マイク入力、リアルタイム文字起こし、翻訳、読み上げを一体化し、医療・接客・学習・旅行などの現場で「相手の発話を聞く、翻訳を見る、必要なら音声で返す」までを少ない操作で完結できることをゴールにする。

重要な方向性は次の通り。

- モバイルで安定して動くことを最優先にする。
- 会話中の待ち時間を短くし、翻訳の途中状態も分かりやすく表示する。
- API キーや会話内容などの取り扱いを安全にする。
- iOS Safari / Android Chrome の差異を前提に、失敗時の復帰手段を用意する。
- Codex で継続的に改善しやすいように、要件、既知課題、検証手順をリポジトリ内に残す。

## 現在の PWA の構成

- `index.html`: メイン UI、設定モーダル、PWA メタタグ。
- `app.js`: 音声認識、OpenAI 翻訳、TTS、エラーレポート、デバウンス最適化をまとめたアプリ本体。
- `style.css`: モバイル向けレイアウトと状態表示。
- `manifest.json`: PWA インストール設定。
- `howto.html`: 使い方。
- `docs/`: TTS 要件、過去の申し送り、Safari 対応メモ。

## 現在の主な問題点

### 1. API キーをブラウザに保存し、直接 OpenAI API を呼んでいる

現在は `localStorage` に OpenAI API キーを保存し、ブラウザから `https://api.openai.com/v1/chat/completions` を直接呼び出す設計になっている。この方式は静的 PWA としては実装が簡単だが、端末内の保存領域やブラウザ拡張、共有端末での扱いに注意が必要になる。

改善策:
- 将来的には軽量なバックエンド、Cloudflare Workers、Vercel Functions などを経由して OpenAI API を呼ぶ。
- ブラウザ側には短期セッション情報だけを持たせ、長期 API キー保存を避ける。
- まずは現行方式を明記し、ユーザーが自分のキーを入れる前提の注意書きを UI と README に追加する。

### 2. アプリ本体が `app.js` 1 ファイルに集中している

音声認識、翻訳、TTS、設定、エラー報告、デバウンス学習が 1 つの大きなファイルにまとまっている。変更時の影響範囲が読みづらく、Codex による段階的改善でも回帰が起きやすい。

改善策:
- `speech-recognition`, `translation`, `tts`, `settings`, `ui-state`, `error-reporting` の責務に分割する。
- まずは動作を変えずに関数境界と状態管理を整理する。
- 重要な状態遷移をテストまたは手動検証チェックリストに落とす。

### 3. PWA としてのオフライン機能が不十分

README にはオフライン機能の記述があるが、現状のファイル一覧では Service Worker が見当たらない。翻訳自体はオンライン API が必要だが、アプリシェルのキャッシュ、起動安定性、更新通知は PWA 体験に直結する。

改善策:
- `sw.js` を追加し、HTML/CSS/JS/アイコンをキャッシュする。
- ネットワーク不通時は「翻訳は利用できないがアプリは起動できる」状態を明示する。
- Service Worker の更新時にユーザーへ再読み込みを促す。

### 4. 旧リポジトリ名や旧 URL が残っている

README やエラーレポートの Issue URL に `Bridge-Ver.4.1-nanoTTS` が残っている。新しいリポジトリへ移行した後は、ユーザー報告やドキュメントが古い場所へ誘導される可能性がある。

改善策:
- README、ドキュメント、エラーレポート URL を `Bridge-TTS-Codex-` に更新する。
- GitHub Pages の公開 URL を決めた後、manifest と README に反映する。

### 5. iOS Safari と Web Speech API の制約が中心リスク

Web Speech API はブラウザ依存が大きく、特に iOS Safari は自動再生、音声認識の継続、TTS 初期化、PWA 起動時の挙動に制約がある。

改善策:
- iOS Safari 用の手動検証手順を固定化する。
- 音声認識が止まったときの再起動、タイムアウト、ユーザー向けメッセージをさらに明確にする。
- TTS は「自動再生」だけに頼らず、タップ再生を常に使える導線にする。

### 6. 翻訳品質評価の仕組みがない

応答速度やデバウンス値の工夫はあるが、翻訳結果の品質を継続的に比較する仕組みがない。モデル変更やプロンプト変更時に改善したかどうかを判断しにくい。

改善策:
- よく使う日英会話サンプルを `docs/evaluation-cases.md` として管理する。
- モデル、プロンプト、温度、レスポンス形式を変えた時に同じケースで比較する。
- 医療・接客など専門領域を扱う場合は、誤訳リスクの高い語句を重点ケースにする。

### 7. UI が会話現場向けにまだ整理できる

現在の UI は機能が多く、設定項目も同じモーダルに集まっている。現場で素早く使うには、通常操作と詳細設定を分離した方がよい。

改善策:
- メイン画面は「入力開始」「停止」「原文」「翻訳」「読み上げ」に絞る。
- API キー、速度、フォント、デバウンス学習は詳細設定にまとめる。
- 状態表示を「聞き取り中」「翻訳中」「読み上げ中」「通信エラー」など会話中に分かる言葉へ統一する。

## 改善ロードマップ

### Phase 0: リポジトリ移行と現状保存

- 新リポジトリへ現行 PWA を初期コミットとしてアップロードする。
- 旧リポジトリ名、旧 URL、既知課題を明文化する。
- これ以降の変更を Codex で追跡しやすい状態にする。

### Phase 1: 安全性と公開準備

- README とエラーレポート URL を新リポジトリへ更新する。
- API キー保存方式の注意書きを UI/README に追加する。
- GitHub Pages の公開設定に合わせて manifest とリンクを更新する。
- Service Worker を追加し、PWA としての最低限のキャッシュを整える。

進捗: 2026-05-18 に `sw.js`、Service Worker 登録、オフライン案内、API キー保存方式の説明、GitHub Pages URL の反映を実施。

### Phase 2: 構造整理

- `app.js` を責務別に分割する。
- 設定読み書き、TTS、翻訳 API 呼び出しを独立したモジュールにする。
- 手動検証チェックリストを作り、変更ごとに確認する。

進捗: 2026-05-18 に [手動検証チェックリスト](MANUAL_VERIFICATION_CHECKLIST.md) を追加。2026-05-21 に `ErrorReporter` を `error-reporter.js` へ、PWA支援処理を `pwa-support.js` へ分離し、`app.js` の責務分割を開始。2026-05-26 にブラウザローカル設定の読み書きを `settings-storage.js` へ、OpenAI 翻訳API通信とストリーミング解析を `translator-service.js` へ分離。2026-05-27 に Playwright スモークテストを追加し、TTS再生管理を `tts-service.js` へ分離。

### Phase 3: 翻訳品質とレスポンス改善

- 翻訳プロンプトを用途別に整理する。
- 評価ケースを作成し、モデルやプロンプト変更時に比較する。
- ストリーミング表示、途中キャンセル、再翻訳の挙動を安定させる。

進捗: 2026-05-29 に [翻訳評価ケース](evaluation-cases.md) を追加し、日常会話、医療・体調、接客・旅行、誤訳リスク重点ケースを比較できる土台を作成。同日に翻訳システムプロンプトを `prompt-service.js` へ分離し、プロンプト改善を独立して扱えるようにした。さらにプロンプトの重要ルールが消えないようにスモークテストへ確認項目を追加。2026-06-02 に翻訳リクエスト ID を導入し、古いストリームや中断済みリクエストが最新の翻訳 UI 状態を上書きしないようにした。同日に `TranslatorService` のストリーム行解析と payload 生成をスモークテストで確認できるようにした。

### Phase 4: 会話現場向け UX 改善

- 片手操作しやすいボタン配置へ調整する。
- 文字サイズ、コントラスト、横画面表示を再設計する。
- 会話ログ、再読み上げ、コピー、履歴消去など実利用で必要な操作を追加する。

進捗: 2026-06-02 に翻訳結果を会話現場で再利用しやすくするため、翻訳ボックスにコピー操作を追加した。

### Phase 5: 本格運用設計

- 必要に応じてバックエンド経由の API 呼び出しへ移行する。
- 利用量制限、エラーログ、匿名診断、プライバシーポリシーを整備する。
- GitHub Pages / Vercel / Cloudflare など運用先を決める。
