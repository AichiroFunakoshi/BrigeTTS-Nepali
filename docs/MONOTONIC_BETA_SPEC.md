# 翻訳方式「順送り（β）」仕様書

対象バージョン: v2.9.1 以降 ／ 状態: **β（試験機能・既定OFF）** ／ 導入PR: #64
設計思想の背景と学術的根拠は [ISHIDATAMI_SCHEME.md](ISHIDATAMI_SCHEME.md)（石畳方式）を参照。

---

## 1. 目的

標準方式（全文再翻訳）と直接A/B比較するための、もう一つの翻訳経路を提供する。

| | 標準（全文再翻訳） | 順送り（β） |
|---|---|---|
| 翻訳トリガー | 認識結果の更新ごと（デバウンス150〜800ms） | 確定セグメント（isFinal）ごと |
| 翻訳対象 | その時点の**全文**を毎回訳し直す | 新しいチャンク**だけ**（文脈を添付） |
| 訳文の表示 | 更新のたびに全体が組み変わり得る | 前から積み上がり、確定分は動かない |
| トークン消費 | 発話が長いほど二次関数的に増える | 発話長に対しほぼ線形＋整形1回 |
| 発話終了時 | 最後の全文翻訳がそのまま確定 | **整形パス**が全体を見直して確定 |
| 語順 | LLM任せ（自然な語順＝逆送りに寄る） | 順送り（原文の情報順）を骨格として保持 |

## 2. 使い方

1. 設定画面 →「翻訳方式」→「順送り（β）」を選択（localStorageに保存・設定エクスポート対象）
2. **切り替えは次の録音から有効**（録音開始時に方式をセッションに固定）
3. 話す → 発話の区切りごとに訳が前から積み上がる
4. 停止 → 整形パスが走り、訳全体が仕上がって確定
5. 確定後は従来どおり: タップでTTS再生・会話履歴保存・大型表示・コピー

## 3. 動作仕様

### 3.1 チャンク訳出パス（録音中）

- **チャンク単位**: 音声認識の確定セグメント（Web版=認識器の `isFinal`、iOSネイティブ=`SpeechRecognitionBridge.swift` の1.3秒ポーズ確定）。日本語入力は既存の `japaneseFormatter` で句読点を整形してから積む
- **文脈の添付**: 各チャンクのuserメッセージに「直前の原文チャンク最大2件＋確定訳の末尾200文字」を『翻訳しないこと』と明記して添える（systemプロンプトは標準方式と共通＝プロンプトキャッシュ維持）
- **直列実行**: チャンクは発話順に1件ずつ翻訳（順序保証のため並列化しない）。完了した訳は確定訳に追記され、**以後再翻訳・再配置しない**（凍結）
- **表示**: `確定訳 + ストリーミング中のチャンク訳`。原文欄は「確定チャンク＋暫定認識」を表示（確定チャンクは認識の自動再起動をまたいで保持される）
- **暫定（interim）認識結果は翻訳しない**（原文欄への表示のみ）。訳出はチャンク確定時

### 3.2 整形パス（録音停止時）

- **トリガー**: 停止ボタン→400ms待機（遅延到着する確定結果を回収）→未訳チャンクの訳出完了を待って実行
- **入力**: 原文全体＋下書き訳（凍結訳の連結）
- **修正するもの**: 誤訳・訳抜け（原文と照合）／音声認識の誤変換の復元／順次訳出に由来する冗長（繰り返し・不要なつなぎ）／フィラー・言い直しの痕跡
- **保持するもの**: 順送りの語順骨格（短文の積み上げ・主要部先出し）。長い前置修飾への再編は禁止。同音・類音で聞き紛れる語の言い換え、句読点＝息継ぎ位置（根拠は ISHIDATAMI_SCHEME.md §3.4）
- **スキップ条件**: チャンクが1件のみの場合（チャンク訳＝全文訳のため。短い発話の応答速度を優先）
- **確定処理**: 整形後の訳で表示更新・`lastTranslationResult` 更新（TTS用）・会話履歴追加・品質チェック・自動読み上げ（会話モード有効時）

### 3.3 エラー・例外時の挙動

| 状況 | 挙動 |
|---|---|
| チャンク翻訳の失敗/タイムアウト（30秒） | エラー表示し次のチャンクへ。訳抜けは整形パスが原文全体から回収 |
| 整形パスの失敗/タイムアウト | **下書き訳をそのまま確定**として使う（訳は消えない） |
| 整形中に遅延チャンクが到着 | 整形完了後に訳出し、再整形して確定し直す |
| 録音再開・リセット | 順送り状態を全消去（世代カウンタで実行中の非同期処理を無効化） |
| リクエスト競合 | チャンク訳・整形は相互排他（並走しない） |

### 3.4 対象外

- レイテンシ計測（F12）は標準方式のみ記録（比較データを汚さないため）
- デバウンス自動最適化の学習データにも順送りセッションは含めない

## 5. 検証

- Playwright smoke: 「supports monotonic translation mode modules」ほか全件（CIで毎PR実行）
- 文脈維持eval: `OPENAI_API_KEY=... node eval/run-eval.js --monotonic`（5ケース: 代名詞・省略主語が直前チャンクの文脈から維持されるか。ケース定義は `eval/cases-monotonic.json`）
- 既存ベースライン: `node eval/run-eval.js`（50ケース。標準方式のプロンプトは無変更のため退行しない設計だが、プロンプト変更時は前後比較すること）

## 6. 既知の制限（β）

1. **ポーズが少ない話者では訳出間隔が長くなる**（チャンク＝確定セグメントのため。標準方式は暫定認識にも追従する）。改善案: 暫定認識の先行訳を薄い表示で併記（未実装）
2. 整形パスの分だけ、停止から確定までに1リクエスト分の時間が加わる（チャンク1件のときはスキップで緩和）
3. 会話履歴には整形後の確定訳のみが入る（途中のチャンク訳は残らない）
4. 順送り訳は主観的には「硬い・不自然」と感じられ得る（客観的理解度はむしろ高い傾向 — ISHIDATAMI_SCHEME.md §2 参照）。A/B比較では「相手に伝わったか」を主に見ること
