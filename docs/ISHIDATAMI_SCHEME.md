# 石畳方式（Ishidatami Scheme）
## 聞き手の実理解を優先する、リアルタイム音声翻訳の二段階訳出スキーマ

**BridgeTTSプロジェクト（Tetsuo ＋ Claude）** ／ 初版: 2026-07-08 ／ 実装: BridgeTTS v2.9.1「順送り（β）」

---

## 概要

リアルタイム日英音声翻訳において、低遅延・表示安定・聞き手の実理解を同時に満たす訳出スキーマを提案する。発話の確定チャンクを原文の順序どおりに訳して**凍結**し（敷いた石は剥がさない）、発話終了時に**語順骨格を保持したまま**誤訳・冗長のみを修正する整形パスを走らせる（目地を整える）。整形で「自然な語順」への再編を意図的に禁止する点が本スキーマの核心であり、その根拠を順送り訳の理解度研究［1–6］と黙読・聴解の音韻処理研究［7–13］に置く。本スキーマは特定アプリに依存しない設計原則として記述し、他のリアルタイム変換系（要約字幕・逐次通訳支援・読み上げ原稿生成等）への流用を想定する。

## 1. はじめに

同時機械翻訳の実装には根本的な緊張がある。速く出すには前から訳すしかないが、前から訳した日本語は「自然な語順」から外れる。逆に、認識更新のたびに全文を訳し直す再翻訳方式（BridgeTTS v2.9.0までの唯一の方式）は自然な訳文を出せるが、訳文が更新のたびに組み変わって読みにくく、発話が長いほど計算量と確定遅延が二次関数的に膨らむ。

このトレードオフを解く鍵を、我々は「訳文の自然さと、聞き手の実理解は別物である」という実証知見［1］に見いだした。石畳方式は、この知見を訳出パイプラインの設計原則に翻訳したものである。

## 2. 背景: 順送り訳は本当に理解を妨げるのか

**順送り訳**（チャンク単位で交差量・省略が0となる訳出［2］）は同時通訳の規範として確立している［4］が、冗長な訳文を生みやすく聞き手の理解を妨げるという懸念が指摘されてきた［2］。繰り返し表現の多用（301回中、名詞の後置修飾起因が約29%［3］）がその典型である。

児島ら［1］はこの懸念を聴解実験で直接検証した（日本語母語話者60名、関係節長3条件×訳タイプ2条件、GLMM分析）。結果は懸念と逆だった:

- **客観的理解度**（内容理解テスト正答率）: 順送り訳が逆送り訳（自然な語順）を上回る傾向（OR=1.67, 95%CI[0.97, 2.86], p=.063。正答率 69.7% vs 59.4%）
- **主観的理解度**: 逆送り訳が高評価（「自然さ」と主観的理解の相関 r=0.61, p<.01）
- **主観と客観はほぼ無相関**（r=0.13）

つまり、自然な語順は「分かった気にさせる」が実際の理解は伴わない——**流暢性の錯覚**である。機構的には、日本語のような主要部後置言語では長い前置修飾が主要部の出現まで作動記憶に保持され続け［6］、これが認知資源を圧迫する。順送り訳は主要部を早期に提示するため保持コストが小さい。読解でも、関係節を後置する順送り訳の読み時間が有意に短いことがアイトラッキングで示されている［5］。

独立した系統の証拠がこれを補強する。黙読中も音韻・構音処理は働き（早口言葉効果［7,8］、言語横断・聾者でも再現［9］）、韻律句と統語構造が一致する文は黙読でも速く読まれる（内在的韻律仮説［10,11］）。句読点は音声の韻律境界と同じ脳反応（CPS）を引き起こす［12］。一方、「文を短く割れば易しくなる」という単純化は理解をかえって下げ得る［13］。これらは「音として処理しやすい形＝係り受けが近く、主要部が早く、息継ぎと意味の区切りが一致する形」が理解を助けることを示しており、順送り訳の優位性と同じ方向を向いている。

## 3. 提案: 石畳方式の4原則

命名の由来: 石畳は、石を進行方向へ一枚ずつ敷き（P2）、敷いた石は剥がさず（P2）、最後に目地を埋めて面を整え（P3）、そして歩く人のためにある（P4＝聞き手の実理解優先）。

### P1. 入口の音韻復元（Sound-in）

音声認識の出力を文字面どおりに信じない。文脈上意味をなさない語は「同じ読み・近い読みの別語」への誤変換とみなし、読みと文脈の両方に合う語へ復元してから訳す（例:「醤油速度」→「処理速度」、「演芸機能」→「嚥下機能」）。入力は音でやってくる、という事実を訳出の前提に置く。

### P2. 石畳訳出（Monotonic Paving）

発話の確定チャンクを原文の順序どおりに訳し、確定訳に追記して**凍結**する。凍結した訳は以後、再翻訳も再配置もしない。文脈は直前チャンクの原文・訳文を翻訳リクエストに添えることで維持する（訳し直しではなく、参照として）。

効果: (a) 遅延が発話長に依存しない、(b) 表示が組み変わらず読者の視線が戻されない、(c) トークン消費が線形になる、(d) 出力が自然と「主要部先出し・短文積み上げ」になり、§2の理解度優位を構造的に獲得する。

### P3. 目地整形（Skeleton-Preserving Refinement）

発話終了時に一度だけ、原文全体と下書き訳を突き合わせて仕上げる。

- **直す**: 誤訳・訳抜け、P1で拾いきれなかった誤変換、順送り特有の冗長（繰り返し・不要なつなぎ）、フィラーの痕跡
- **直さない**: 語順の骨格。長い前置修飾への再編は禁止。一文が長くなるなら再編ではなく分割する

この非対称性が本スキーマの核心である。整形を無制約のLLMに任せると「自然な語順」＝逆送りへ引き戻され、主観評価と引き換えに実理解を失う（§2の流暢性の錯覚）。**整形とは骨格を保ったまま目地を埋めることであり、石を敷き直すことではない。**

### P4. 耳優先の出力方針（Ear-First Output）

確定訳は音声（TTS）または画面で相手に提示される。どちらの経路でも音韻処理が働く（§2）ため:

- 聞き紛れやすい同音・類音の語は、文脈で紛れない語に言い換える（正確さを損なわない範囲で）
- 句読点は息継ぎの位置＝意味の区切りに打つ（TTSのポーズ制御を兼ねる）
- 数値・固有名詞・専門用語の正確さはリズムに常に優先する

## 4. 実装（BridgeTTS v2.9.1）

2つのLLMコールで実現する。チャンク訳（P1+P2）は既存の翻訳systemプロンプト（音韻復元則を含む）＋文脈付きuserメッセージ、整形（P3+P4）は専用systemプロンプト（`prompt-service.js` `refinementSystemPrompt`）。整形失敗時は下書き訳を確定として使い、訳を消さない。詳細仕様は [MONOTONIC_BETA_SPEC.md](MONOTONIC_BETA_SPEC.md)。

検証: Playwrightスモーク13件、文脈維持eval 5ケース（`eval/cases-monotonic.json`: 代名詞・省略主語の維持）、既存50ケースの品質ベースラインは無変更経路のため不変。

## 5. 考察と限界

1. ［1］の主効果はp=.063の「傾向」であり、材料は関係節を含む物語文に限られる。医療対話など本アプリの実使用域での外的妥当性は、標準方式とのA/B比較（本実装の目的）で確かめる
2. 順送り訳は主観的には低評価になり得る［1］。ユーザー評価を「自然に感じたか」で取ると本スキーマは不利に見える。**評価は「相手に正しく伝わったか」で行うこと**
3. チャンク＝ポーズ単位のため、ポーズの少ない話者では訳出間隔が延びる。暫定認識の投機的先行訳の併記は将来課題
4. P3の「骨格保持」はプロンプト制約であり、LLMが常に従う保証はない。逸脱の検出（訳文チャンクの交差量測定［2］の自動化）は将来課題

## 6. 他システムへの流用チェックリスト

石畳方式は「ストリーム入力 → 逐次変換 → 確定時整形」の構造を持つ系に一般化できる（例: リアルタイム要約字幕、会議議事録の逐次生成、逐次読み上げ原稿変換）。流用時は次を確認する:

- [ ] 入力は音声・OCR等の**ノイズを含むストリーム**か → P1（入口の復元）を系に合わせて定義
- [ ] 逐次出力の**凍結単位**（石）は何か → 意味の区切り＝ポーズ・句読点・段落
- [ ] 整形で**保持すべき骨格**は何か → 「提示順序」が受け手の処理コストを下げている構造（本件では語順）を特定し、整形プロンプトで明示的に保護する
- [ ] 受け手は**耳・目のどちらで、どんな時間制約で**受け取るか → P4を媒体に合わせて調整
- [ ] 評価は**受け手の実理解**で行う（作り手・受け手の主観的流暢性で測らない）
